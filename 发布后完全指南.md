## 第三部分：如何与其他开发者互动

### 📬 接收和处理 Issue

#### 1. 查看 Bug 报告

**路径**：GitHub 仓库 → Issues → 筛选 "bug" label

**你会看到**：
```
🐛 [BUG] 创建实例时报错
#1 opened 2 hours ago by user123
```

**处理流程**：
1. 点击 Issue 查看详情
2. 阅读 Bug 描述和复现步骤
3. 尝试复现问题
4. 回复确认/询问更多信息
5. 添加 label（如 `confirmed`, `help wanted`）
6. 修复后回复并关闭 Issue

**常用回复模板**：
```markdown
感谢反馈！我已经复现了这个问题。

**原因**：XXX
**修复计划**：将在 v1.6.1 中修复
**临时解决方案**：XXX

再次感谢！
```

---

#### 2. 查看功能建议

**路径**：GitHub 仓库 → Issues → 筛选 "enhancement" label

**处理流程**：
1. 阅读功能描述和动机
2. 评估是否合理/可行
3. 回复接受/拒绝/需要更多信息
4. 如果接受，添加到 TODO 列表
5. 实现后关闭 Issue

**接受回复模板**：
```markdown
很好的建议！这个功能确实有用。

**计划**：加入 v1.7.0 开发计划
**优先级**：中/高
**预计时间**：X 周内

欢迎贡献代码！
```

**拒绝回复模板**（委婉）：
```markdown
感谢建议！不过目前这个功能与项目核心目标不太符合。

**原因**：XXX
**替代方案**：可以考虑 XXX

希望你能理解，欢迎继续提建议！
```

---

### 🔀 接收 Pull Request（PR）

#### 1. 查看 PR

**路径**：GitHub 仓库 → Pull requests

**你会看到**：
```
feat: 添加部门导出功能
#5 opened 1 day ago by contributor
```

#### 2. 审查代码流程

**步骤**：
1. 点击 PR 查看详情
2. 查看 **Files changed** 标签
3. 检查代码质量：
   - 是否符合代码规范
   - 是否有测试
   - 是否有文档更新
4. 在代码行上评论（如果有问题）
5. 选择：
   - ✅ **Merge** - 合并
   - 📝 **Request changes** - 要求修改
   - 👀 **Comment** - 仅评论

#### 3. 合并 PR

**如果代码 OK**：
1. 点击 **Merge pull request**
2. 选择合并方式（推荐 Squash and merge）
3. 确认合并

**合并后回复**：
```markdown
已合并！感谢贡献 🎉

这个功能会在 v1.7.0 中发布。
```

---

### 🏷️ 使用 Label 管理 Issue/PR

**常用 Label**：

| Label | 含义 | 使用场景 |
|-------|------|---------|
| `bug` | Bug | 确认是 Bug 的 Issue |
| `enhancement` | 功能增强 | 功能建议 |
| `good first issue` | 适合新手 | 简单任务，鼓励新人参与 |
| `help wanted` | 需要帮助 | 需要社区贡献 |
| `duplicate` | 重复 | 与已有 Issue 重复 |
| `wontfix` | 不修复 | 决定不处理 |
| `question` | 问题 | 需要更多信息 |
| `documentation` | 文档 | 文档相关 |

**添加 Label**：
1. 打开 Issue/PR
2. 右侧点击 Labels
3. 选择一个或多个

#### 1. 保持友好和专业

**好**：
```

```

**不好**：

#### 2. 及时回复

- **Bug 报告**：24-48 小时内回复

如果暂时无法处理：
```markdown

```

#### 3. 鼓励贡献


### Q: 有人提交了 Bug，但我无法复现怎么办？

A: 
```markdown
感谢报告！不过我无法复现这个问题。

能否提供：
1. 详细的复现步骤
2. 系统信息（OS/版本）
3. 日志文件
4. 截图/录屏

这样我可以更好地定位问题。
```
