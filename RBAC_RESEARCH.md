## 一、类似系统权限设计对比

### 1. Grafana（监控仪表盘）⭐⭐⭐⭐⭐

#### 角色体系
```
三层权限模型：

1️⃣ 服务器管理员（Server Admin）
   - 管理所有组织
   - 管理用户和许可证
   - 查看服务器统计

2️⃣ 组织管理员（Org Admin）
   - 管理组织内所有资源
   - 管理用户和团队
   - 管理数据源

3️⃣ 编辑器（Editor）
   - 查看和编辑仪表盘
   - 管理文件夹
   - 查询数据源

4️⃣ 查看员（Viewer）
   - 只能查看仪表盘
   - 不能编辑
   - 可以查询数据
```

#### 关键设计
| 特性 | 实现方式 | 可借鉴 |
|------|----------|--------|
| **权限继承** | 组织权限 → 仪表盘权限 | ✅ |
| **权限覆盖** | 仪表盘权限可覆盖组织权限 | ✅ |
| **团队概念** | 团队作为权限边界 | ✅ |
| **资源隔离** | 按组织隔离数据 | ✅ |

#### 适用场景
- ✅ 多租户 SaaS
- ✅ 多部门企业
- ✅ 需要数据隔离

---

### 2. Portainer（Docker/K8s 管理）⭐⭐⭐⭐

#### 角色体系
```
四层权限模型：

1️⃣ 管理员（Administrator）
   - 管理所有环境
   - 管理用户和团队
   - 管理注册表

2️⃣ 帮助台（Helpdesk）
   - 查看环境
   - 重启容器
   - 查看日志

3️⃣ 运维人员（Operator）
   - 管理容器
   - 管理镜像
   - 管理网络

4️⃣ 普通用户（User）
   - 管理自己的容器
   - 查看分配的环境
```

#### 关键设计
| 特性 | 实现方式 | 可借鉴 |
|------|----------|--------|
| **环境隔离** | 按环境（Environment）分组 | ✅ |
| **团队授权** | 权限赋予团队而非个人 | ✅ |
| **操作审计** | 记录所有操作日志 | ✅ |
| **只读角色** | 帮助台角色（只能看和重启） | ✅ |

#### 适用场景
- ✅ IT 运维管理
- ✅ 多环境管理
- ✅ 需要操作审计

### 3. Kubernetes（容器编排）⭐⭐⭐⭐⭐

#### RBAC 模型
```
四个核心对象：

1️⃣ Role（角色）
   - 定义权限集合
   - 作用域：单个 Namespace

2️⃣ ClusterRole（集群角色）
   - 定义权限集合
   - 作用域：整个集群

3️⃣ RoleBinding（角色绑定）
   - 绑定用户到 Role
   - 作用域：单个 Namespace

4️⃣ ClusterRoleBinding（集群角色绑定）
   - 绑定用户到 ClusterRole
   - 作用域：整个集群
```

#### 权限定义
```yaml
# Role 示例
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: default
  name: pod-reader
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "watch", "list"]
```

#### 关键设计
| 特性 | 实现方式 | 可借鉴 |
|------|----------|--------|
| **声明式权限** | YAML 定义权限 | ⚠️ 太重 |
| **细粒度控制** | 精确到 API 动词 | ⚠️ 太复杂 |
| **命名空间隔离** | Namespace 隔离 | ✅ |
| **角色复用** | ClusterRole 跨 Namespace | ✅ |

#### 适用场景
- ❌ 不适合小型应用（太重）
- ✅ 适合大型平台
- ✅ 需要极细粒度控制

## 二、权限模型总结

### 常见角色类型

| 角色类型 | 职责 | 典型权限 |
|----------|------|----------|
| **超级管理员** | 系统管理 | 所有权限 + 用户管理 |
| **组织管理员** | 组织/部门管理 | 组织内所有权限 |
| **运维管理员** | 日常运维 | 部署、监控、配置 |
| **编辑器** | 内容编辑 | 编辑权限 |
| **查看员** | 数据查看 | 只读权限 |
| **审计员** | 审计合规 | 审计日志 + 报表 |

---

### 权限边界设计

#### 方案 1：按组织隔离（Grafana）
```
┌─────────────────────────────────┐
│         服务器                   │
│  ┌─────────┐  ┌─────────┐      │
│  │ 组织 A   │  │ 组织 B   │      │
│  │ - 用户   │  │ - 用户   │      │
│  │ - 仪表盘 │  │ - 仪表盘 │      │
│  └─────────┘  └─────────┘      │
└─────────────────────────────────┘
```

**适用**: 多租户 SaaS、多部门企业

#### 方案 2：按环境隔离（Portainer）
```
┌─────────────────────────────────┐
│         平台                     │
│  ┌─────────┐  ┌─────────┐      │
│  │ 环境 A   │  │ 环境 B   │      │
│  │ - 容器   │  │ - 容器   │      │
│  │ - 镜像   │  │ - 镜像   │      │
│  └─────────┘  └─────────┘      │
└─────────────────────────────────┘
```

**适用**: IT 运维、多环境管理

#### 方案 3：按资源隔离（Kubernetes）
```
┌─────────────────────────────────┐
│         集群                     │
│  ┌─────────┐  ┌─────────┐      │
│  │ NS A    │  │ NS B    │      │
│  │ - Pods  │  │ - Pods  │      │
│  │ - SVC   │  │ - SVC   │      │
│  └─────────┘  └─────────┘      │
└─────────────────────────────────┘
```

**适用**: 大型平台、需要细粒度控制

### 数据隔离方案

#### 方案 A：部门隔离（推荐）⭐
```
部门作为数据边界：
- 实例归属部门
- Token 消耗按部门统计
- 查看员只能看到本部门数据
- 管理员可以看到所有部门
```

#### 方案 B：项目隔离
```
项目作为数据边界：
- 实例归属项目
- 成本按项目核算
- 项目成员只能看到本项目数据
```

#### 方案 C：标签隔离
```
标签作为数据边界：
- 实例打标签（部门/项目/环境）
- 按标签过滤数据
- 灵活但实现复杂
```

**推荐**: 方案 A（部门隔离）- 最符合企业实际使用场景

## 五、技术选型建议

### 认证方案

| 方案 | 优点 | 缺点 | 推荐 |
|------|------|------|------|
| **JWT** | 无状态、易扩展 | 需要处理刷新 | ✅ 推荐 |
| **Session** | 简单、易实现 | 需要存储 | ⚠️ 可选 |
| **OAuth2** | 支持第三方登录 | 实现复杂 | ❌ 暂不 |

### 权限存储

| 方案 | 优点 | 缺点 | 推荐 |
|------|------|------|------|
| **数据库** | 灵活、可查询 | 需要设计表结构 | ✅ 推荐 |
| **配置文件** | 简单 | 不灵活 | ❌ 不推荐 |
| **LDAP/AD** | 企业集成 | 实现复杂 | ❌ 暂不 |

### 推荐技术栈

```
认证：JWT + Refresh Token
权限存储：SQLite/PostgreSQL（现有数据库）
前端权限：路由守卫 + 按钮级权限
后端权限：中间件验证
```

## 六、数据库设计建议

### 用户表
```sql
CREATE TABLE users (
    id TEXT PRIMARY KEY,
    username TEXT UNIQUE NOT NULL,
    password_hash TEXT NOT NULL,
    role TEXT NOT NULL,           -- 'admin' | 'operator' | 'viewer' | 'auditor'
    department_id TEXT,           -- 所属部门
    created_at TEXT NOT NULL,
    last_login_at TEXT
);
```

### 部门表
```sql
CREATE TABLE departments (
    id TEXT PRIMARY KEY,
    name TEXT UNIQUE NOT NULL,
    description TEXT,
    created_at TEXT NOT NULL
);
```

### 实例 - 部门关联表
```sql
CREATE TABLE instance_departments (
    instance_id TEXT NOT NULL,
    department_id TEXT NOT NULL,
    PRIMARY KEY (instance_id, department_id)
);
```

## 七、总结

### 最佳实践总结

1. **角色不宜过多** - 4-5 个基础角色足够
2. **权限继承** - 上级角色自动拥有下级权限
3. **数据隔离** - 按部门/组织隔离数据
4. **操作审计** - 记录所有关键操作
5. **最小权限** - 默认拒绝，只授予必要权限

### OpenCLAW Manager 推荐方案

```
✅ 4 个基础角色（超级管理员/运维/查看员/审计员）
✅ 部门数据隔离
✅ JWT 认证
✅ 数据库存储权限
✅ 操作审计日志
✅ 分阶段实现（v1.0 基础 → v1.5 部门 → v2.0 完整 RBAC）
```
