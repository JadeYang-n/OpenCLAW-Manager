# 端口配置最佳实践

这是一个关于前后端端口配置的最佳实践指南。它帮助开发者避免常见的端口配置错误，特别是在前后端分离的项目中。

## 🚨 严重问题：端口不一致

如果用户报告了类似错误：
- `ERR_CONNECTION_REFUSED` on port 8082 but server runs on 8080
- `Failed to fetch` on login
- Network panel shows wrong port

**立即执行端口一致性检查：**

1. **搜索所有端口引用**
   ```bash
   # 在前端代码库中搜索
   grep -r "localhost:808" --include="*.ts" --include="*.tsx"
   
   # 检查后端配置
   grep -r "port:" --include="settings.rs" --include="*.toml"
   ```

2. **检查以下关键文件**
   - 前端 API 客户端 (services/api.ts, lib/api.ts)
   - 模块特定 API (modules/*/api.ts)
   - 硬编码的 fetch 调用
   - 环境变量文件 (.env, .env.local)

## ✅ 最佳实践清单

### 1. 端口配置统一
- [ ] 所有前端 API 客户端使用同一个 base URL 常量
- [ ] 端口号应在 .env 中定义，不在代码中硬编码
- [ ] 后端配置文件单独管理端口

### 2. 代码审查检查点
- [ ] grep 搜索所有端口引用
- [ ] 验证前端构建后的 dist/index.html
- [ ] 浏览器 Network 面板验证实际请求

### 3. 测试验证
- [ ] 登录功能实际测试
- [ ] 所有 API 端点测试
- [ ] Network 面板检查所有请求端口

## 📝 工作流铁律执行

根据《项目开发总纲》：

```
1. 读代码
   ├─ 搜索所有端口引用 (grep -r)
   ├─ 检查 services/api.ts
   ├─ 检查 lib/api.ts  └─ 检查所有 fetch() 调用

2. 最小修改
   └─ 只改端口号，不改逻辑

3. 验证
   ├─ grep 确认没有遗漏
   └─ 浏览器 Network 面板验证
```

## 🛠️ 快速诊断

**症状：前端报告 ERR_CONNECTION_REFUSED**

1. 打开浏览器开发者工具 → Network 面板
2. 尝试触发错误的操作（如登录）
3. 查看失败请求的 URL
4. 检查 URL 中的端口 vs 实际后端端口
5. 如果不一致：
   - 搜索所有端口引用
   - 统一改为正确的端口
   - 重新构建/刷新

## 💡 关键要点

- **绝对不要**在多个文件中硬编码相同的端口号
- **总是**使用环境变量或统一的配置文件
- **总是**使用浏览器开发者工具验证实际请求
- **总是**在修改前使用 grep 搜索

## 🛠️ 故障排除

如果问题仍然存在，请检查：
1. 浏览器缓存是否清除 (Ctrl+Shift+R)
2. 后端服务是否正在运行
3. 防火墙是否阻止了端口
4. 是否有多个后端实例在不同端口运行

## 📖 经验教训（真实案例）

### 问题描述
用户报告前端显示 `POST http://localhost:8082/api/v1/users/login net::ERR_CONNECTION_REFUSED`，但后端实际运行在 8080 端口。

### 根本原因
1. 前端 `lib/api.ts` 使用的是 8082 端口
2. 前端 `TokenAnalysisPage.tsx` 使用的也是 8082 端口
3. `services/api.ts` 使用的是 8080 端口
4. 端口不一致导致连接错误

### 修复过程
1. 搜索所有端口引用：`grep -r "8082" --include="*.ts" --include="*.tsx"`
2. 发现 problematic 文件：
   - `apps/web/src/lib/api.ts` (第1行)
   - `apps/web/src/modules/token/TokenAnalysisPage.tsx` (第91行)
3. 将所有 8082 改为 8080
4. 重新构建前端
5. 刷新浏览器验证

### 教训总结
- **永远不要**在多个文件中硬编码相同的端口号
- **总是先执行全局搜索**再做修改
- **总是使用浏览器开发者工具**验证实际请求
- **遵循工作流铁律**：读代码 → 最小修改 → 验证

### 正确的做法
```bash
# 1. 搜索所有端口引用
grep -r "localhost:808" --include="*.ts" --include="*.tsx" apps/web/src/

# 2. 检查配置文件
grep -r "port:" --include="settings.rs" src/server/src/config/

# 3. 创建统一的配置
# 在 apps/web/.env 中添加
VITE_API_PORT=8080

# 在 services/api.ts 中使用
const API_BASE_URL = `http://localhost:${import.meta.env.VITE_API_PORT}/api/v1`
```
